# ctx

A context layer for multi-repo workspaces. Maps features across repositories with product definitions, engineering contracts, and implementation docs ‚Äî designed for AI-assisted development.

## What it does

ctx sits on top of your repos and gives AI agents (and humans) a structured way to understand features that span multiple services. Each feature gets three files:

- **product.md** ‚Äî what the feature does, use cases, business rules
- **engineering.md** ‚Äî technical contracts, key interfaces, constraints
- **implementation.md** ‚Äî how it's actually built today, key files, flows

Skills use these files to implement new use cases, review contracts, and map existing code ‚Äî all without losing context across repos.

## Install

```bash
npx skills add nicolasse/ctx
```

## Getting started

```bash
# 1. Create a workspace directory
mkdir my-project && cd my-project

# 2. Clone your repos at the root level
git clone git@github.com:your-org/repo-a.git
git clone git@github.com:your-org/repo-b.git

# 3. Open Claude Code and init
claude
# then: /ctx-init
```

`/ctx-init` asks for your project name, sets up the workspace structure, explores all repos, and generates a workspace map.

## Skills

| Skill | What it does |
|---|---|
| `/ctx-init` | Name the project, set up workspace, explore repos, generate workspace map |
| `/ctx-export` | Scan repos and export `context/config.json` with repo list and context link |
| `/ctx-pull-repos` | Clone repos from `config.json` into the workspace root |
| `/ctx-add-feature` | Design and implement a new feature from scratch |
| `/ctx-add-use-case` | Add a use case to an existing feature with contract validation |
| `/ctx-map-product` | Explores repos + talks to you ‚Üí generates `product.md` |
| `/ctx-map-engineering` | Explores repos ‚Üí extracts technical contracts into `engineering.md` |
| `/ctx-map-implementation` | Explores repos ‚Üí maps current implementation into `implementation.md` |

## Workspace structure

```
my-project/
  context/                   ‚Üê project context (its own git)
    WORKSPACE.md             ‚Üê generated by ctx-init
    config.json              ‚Üê repo list for ctx-pull-repos
    follow-up/               ‚Üê feature context
      product.md
      engineering.md
      implementation.md
  repo-a/                    ‚Üê repos live at the root
  repo-b/
```

## Workflow

**First time:**
1. `/ctx-init` ‚Äî names project, explores repos, generates `WORKSPACE.md`
2. `/ctx-export` ‚Äî scans repos and generates `context/config.json`

**Existing feature (already in code, needs context files):**
1. `/ctx-map-product` ‚Üí generates `product.md`
2. `/ctx-map-engineering` ‚Üí generates `engineering.md`
3. `/ctx-map-implementation` ‚Üí generates `implementation.md`

**New feature (doesn't exist in code):**
1. `/ctx-add-feature` ‚Äî designs product + engineering, implements, validates, documents

**Adding behavior to an existing feature:**
1. `/ctx-add-use-case` ‚Äî implements, validates contracts, updates context files

**Sharing the workspace:**
1. `/ctx-export` ‚Äî generates `context/config.json` with all repo URLs
2. Push the `context/` repo to GitHub
3. Others clone it and run `/ctx-pull-repos` to get all repos
